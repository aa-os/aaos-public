# M15 Track D cross-control regression contract

Track D is a deterministic, offline, read-only, additive, contract-first, simulation-only regression package. It composes caller-supplied inert mappings to prove that evidence introduced by M15 Tracks A-C cannot bypass independently governed AAOS controls. It imports or executes no referenced artifact and changes no Track A, B, or C contract.

## Governing separations

Learning Proof is not retention approval, training approval, memory persistence, skill admission, runtime execution, or Learning Proof sealing. Capability Memory Pack is not package admission, tool registration, executable capability, or runtime authorization. Rollback readiness is not rollback authorization or execution. Deletion evidence is not deletion authorization, physical erasure, or provider-side deletion. Portability evidence is not provider migration approval, replacement-model authorization, or production execution.

AI provenance is not identity proof, human review, or merge approval. Public evidence is not public-output authorization or permission to expose sensitive material. Decision Proof linkage is not verification or sealing. Learning Proof sealing and Decision Proof sealing remain AAOS-owned. AAOS remains the decision sovereignty layer.

## Control domains

The machine-readable matrix binds Tracks A-C and cross-track composition to public-output safety; provenance and human review; skill, MCP package, tool, and capability admission; evidence extraction, freshness, supersession, and revocation; governance and execution authority; Decision Proof ownership; and M15/release publication state. Each binding records the maintained source PR, exact repository paths, source contract or schema version, source boundary statements, expected source output classes, maintained-main path binding, and the adapter test that executes the source control.

The bound controls are the M14 public-output exfiltration gate from PR #204, M14 AI provenance and human-review routing from PR #206, M14 external skill admission from PR #208, external evidence admission from PR #209, and the M14 cross-control authority boundary from PR #210. Track D consumes only JSON-compatible inert output snapshots produced by their existing pure evaluators in tests. Every composition requires outputs for all five bound controls; an omitted output rejects, so lexical evaluation can never become the sole composition proof. Snapshotting removes runtime object identity; cyclic or shared identities presented directly to Track D remain malformed and reject. Track D neither executes the provenance workflow nor treats fixture validity, routing, readiness, admission eligibility, or evidence accumulation as approval. Lexical scanning remains a supplemental adversarial guard, not the cross-control proof. The PR #206 provenance and PR #210 cross-control contracts expose no native version field, so their bindings state `unversioned-contract` explicitly and combine the source PR with maintained-main path binding instead of fabricating a fixture version.

Sensitive-output inspection applies recursively to every mapping key and scalar value in the complete caller-supplied document, including extensions, evidence references, identifiers, reason codes, review and provenance metadata, graph nodes, and capability manifests. Schema validity, trusted workflow origin, Decision Proof linkage, synthetic classification, verification, and portability never exempt inspection. Findings identify locations and categories but never echo detected values. Structured category markers are recognized in both field names and scalar content; fixture 02 isolates synthetic confidential-prompt content under a generic payload field, while fixture 17 remains the credential-shaped-content case.

Track D preserves category and location, then adapts confidential prompts, hidden reasoning or traces, personal and patient identifiers, medical and financial identifiers, private evaluations, provider-managed memory, production accounts, internal paths, private configurations, cross-repository private summaries, and credential-shaped values into the existing M14 gate's five structured input classes. The existing M14 gate remains the blocking control; Track D does not duplicate or weaken its authorization semantics.

## Evaluation contract

The evaluator accepts caller-supplied inert mappings only. It performs no network access, file access, subprocess or shell execution, dynamic external import, package installation, MCP registration, tool execution, model invocation, file mutation, rollback, deletion, provider operation, or external state transition. It does not mutate input. Cyclic containers, shared container identities, nesting-limit violations, container-limit violations, and hostile mappings are explicit malformed findings and always reject before source adapters are consumed. No malformed branch is silently omitted. Findings are deterministic, sorted, and deduplicated.

`pass` means only that the supplied synthetic evidence preserved the tested boundaries. `quarantine` is used for sensitive-output findings. `reject` is used for all other boundary violations. No result grants governance, policy, identity, risk, deployment, installation, package admission, tool registration, runtime, rollback, deletion, migration, replacement-model, audit, waiver, sealing, milestone-completion, or release authority.

## Release boundary

Track D does not close tracker #231, declare M15 complete, approve completion review, finalize release readiness, authorize v0.14.0, create a tag, or create a GitHub Release. Tracker #231 remains Open. M15 remains active and incomplete. v0.14.0 remains unpublished.

All fixtures are synthetic and contain no real credentials, tokens, personal information, medical or financial identifiers, confidential prompts, private evaluations, provider-managed memory, or production account information.
